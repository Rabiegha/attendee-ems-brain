# Load tests — L9.b après réservation tardive — diagnostic du palier 50/s

## Verdict rapide

- Staging uniquement, backend `38d8a2138229f9b46107bea15b4eb16c035d731b`.
- **25 inscriptions/s pendant deux minutes : validé** avec 3 001/3 001 confirmations, aucune
  perte et p95 128,14 ms.
- **50 inscriptions/s avec L9.a actif : non validé** sur ce SHA. Un essai court peut passer,
  mais le débit soutenu recrée une file et des commits après abandon du client.
- Le test discriminant avec `registration_auto_approve=false` absorbe 50/s : il localise la
  dégradation dans la combinaison du verrou de session et du compteur événement L9.a, et non dans
  k6, la mémoire ou une saturation permanente de PostgreSQL.

## Cohérence et risque UX

- Chaque run termine sans surbooking et avec `Session.registered_count` égal au nombre réel de
  choix confirmed.
- Le run 25/s donne exactement 3 001 inscriptions, 3 001 emails distincts et 3 001 choix.
- Lors du premier échec 50/s, PostgreSQL contient finalement 266 inscriptions alors que le client
  n'a reçu que 126 confirmations. Lors du retry, 789 inscriptions sont committées pour 633 réponses
  reçues.
- Ce delta confirme le risque UX : interrompre k6 ou expirer côté client ne signifie pas que la
  transaction serveur a été annulée. Une relance doit rester idempotente et l'interface ne doit pas
  annoncer un échec définitif sans réconciliation.

## Diagnostic technique

- L'observation des verrous toutes les 500 ms n'a vu qu'un bref wait `transactionid`, au maximum
  neuf transactions actives et aucune longue transaction persistante.
- Le palier 25/s a atteint environ 167 % CPU API et 80 % CPU PostgreSQL, sans pression mémoire,
  restart ni OOM ; le générateur n'est pas la limite à ces débits.
- Lorsque L9.a est neutralisé uniquement sur la fixture, 50/s redevient stable et rapide.
- Cause retenue : la route nominale verrouille la même ligne `sessions`, puis conserve ce verrou
  pendant la création du choix et le commit incluant le trigger différé L9.a. Les requêtes suivantes
  s'empilent sur ce point chaud.

## Décision et correction attendue

- Ne pas déclarer 50/s ni 3 000 simultanées sur ce SHA.
- Fusionner uniquement après régressions la correction qui regroupe réservation du siège et création
  du choix dans une instruction SQL atomique, puis rejouer smoke, 25/s et 50/s complet.
- Conserver L9.a actif dans toute mesure finale : le run sans auto-approve est un test de diagnostic,
  pas l'architecture LFD cible.

## Traçabilité

- Artefacts bruts : `attendee-ems-back/tmp/k6/<run>/gen-01/` pour chacun des sept identifiants
  ci-dessus.
- Invariants contrôlés après chaque run : fixture, portée event/org/session, unicité email et
  participant, statut confirmed, compteur global exact, capacité et absence de surbooking.
