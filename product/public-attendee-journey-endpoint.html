Endpoint public — Parcours d'un participant (Attendee Journey)

Endpoint conçu pour qu'une application externe (scoring, gamification, dashboard partenaire) puisse récupérer le parcours individuel d'un participant à partir de la valeur encodée dans son QR code de badge.

1. Spécification

GET /api/public/events/:publicToken/attendees/:badgeRef/journey

AuthAucune — endpoint public
CORSAccess-Control-Allow-Origin: * (cohérent avec /live-stats)
Tag SwaggerPublic

Paramètres d'URL

ParamTypeDescription
publicTokenstringToken public unique de l'événement (event_settings.public_token)
badgeRefUUIDValeur encodée dans le QR du badge = registration.id

Codes de réponse

CodeCas
200Parcours retourné
404Événement introuvable ou participant introuvable / hors événement / soft-deleted / badgeRef non UUID
Pour des raisons de sécurité, on ne distingue pas "événement introuvable" de "participant introuvable" côté client : les deux renvoient 404 afin d'éviter l'énumération.

Sémantique des champs

ChampDescription
attendee.first_name / last_name / companyIdentité du participant. En priorité depuis attendees.*, fallback sur registrations.snapshot_*. null si non renseigné.
totals.partnersNombre DISTINCT de partenaires ayant scanné ce participant sur cet événement. Calculé sur partner_scans, distinct sur scanned_by, hors soft-deletes.
sessions[]Liste chronologique des sessions où le participant a été scanné IN au moins une fois. Une seule entrée par session (premier scan IN).
sessions[].session_idUUID de la session. À matcher côté front avec la liste fixe d'IDs spéciales pour calculer is_special.
sessions[].session_nameNom courant de la session (non snapshot).
sessions[].scanned_atTimestamp ISO-8601 UTC du premier scan IN.
⚠️ is_special n'est pas retourné : la classification est gérée côté front à partir d'une liste fixe d'IDs. L'endpoint reste générique.

3. Exemples d'appel

cURL

curl -s "https://api.attendee.example.com/api/public/events/lfd-2026-public-token/attendees/3f1e2a4b-7c91-4d8e-9b22-1a5e8c0d6f44/journey" | jq

TypeScript / fetch

async function fetchAttendeeJourney(publicToken, badgeRef) {
  const r = await fetch(
    `https://api.attendee.example.com/api/public/events/${publicToken}/attendees/${badgeRef}/journey`,
  );
  if (r.status === 404) throw new Error('Event or attendee not found');
  if (!r.ok) throw new Error(`HTTP ${r.status}`);
  return r.json();
}

Calcul de score côté client

const SPECIAL_SESSION_IDS = new Set([
  '7f3c5d2b-9b1a-4b58-9f1e-1c4a6d7e2b10',
  '0c2d18e3-2efb-4f9a-9b34-2a6f3c1d5b88',
  'aa11bb22-cc33-dd44-ee55-66778899aabb',
]);

function score(journey) {
  const partners = journey.totals.partners;
  const special = journey.sessions.filter(s => SPECIAL_SESSION_IDS.has(s.session_id)).length;
  const normal = journey.sessions.length - special;
  return partners * 10 + normal * 5 + special * 20;
}

4. Architecture & implémentation

ÉlémentEmplacement
Routesrc/modules/public/public.controller.ts
Logique métierPublicService.getAttendeeJourney()
DTO de réponsesrc/modules/public/dto/attendee-journey.dto.ts
ModulePublicModule (déjà importé dans AppModule)

Garanties multi-tenant

  1. publicToken résolu uniquement via event_settings.public_token (pas de fallback par UUID brute).
  2. registration récupérée avec event_id + org_id issus de l'event résolu — cross-event impossible.
  3. partner_scans filtrés sur org_id + event_id + registration_id.
  4. session_scans filtrés par la relation session.event_id + session.org_id.
  5. Soft-deletes respectés : registrations.deleted_at IS NULL, partner_scans.deleted_at IS NULL.

Validation d'entrée

badgeRef est validé contre une regex UUID v1-v5 avant tout accès Prisma. Un format invalide renvoie 404 (pas 400) pour ne pas exposer la structure interne.

Performances

  • Maximum 4 requêtes Prisma indexées (event_settings, registration, partner_scans distinct, session_scans).
  • Aucune jointure lourde. Réponse typiquement < 50 ms.
  • Compatible avec une mise en cache HTTP courte (5–10 s) côté CDN si besoin.

6. Checklist QA

  • publicToken inexistant → 404
  • badgeRef non UUID → 404
  • badgeRef d'un autre événement → 404 (isolation cross-event)
  • registration.deleted_at != null → 404
  • Aucun partenaire n'a scanné → totals.partners = 0
  • Même partenaire scanne 3 fois → totals.partners = 1 (distinct)
  • Plusieurs scans IN sur la même session → 1 seule entrée, scanned_at = premier scan
  • Scans OUT ignorés
  • Header Access-Control-Allow-Origin: * présent
  • Swagger visible sur /api/docs (tag Public)
